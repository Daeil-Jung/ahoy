## 핵심 아키텍처

### 설계 철학

"에이전트 평가의 표준화" — 정확도만이 아닌 **비용-성능 트레이드오프**를 기본 포함. 제3자 플랫폼으로서 공정한 에이전트 비교 제공.

### 2대 핵심 컴포넌트

1. **중앙 리더보드 플랫폼**: 비용 통제 평가를 기본 포함, 비용-성능 트레이드오프 가시화
2. **표준화된 평가 하네스**: 재현 가능한 에이전트 평가, 토큰 사용량/트레이스 자동 추적

### 지원 벤치마크

| 벤치마크 | 도메인 |
|----------|--------|
| SWE-bench Verified | 소프트웨어 엔지니어링 |
| USACO | 알고리즘/프로그래밍 대회 |
| AppWorld | 애플리케이션 개발 |
| CORE-bench | 핵심 코딩 능력 |
| tau-bench | 범용 에이전트 태스크 |

### 신뢰성 대시보드 (Reliability Dashboard)

정확도 이상의 **다차원 에이전트 행동 평가**:
- **일관성 (Consistency)**: 동일 입력에 대한 반복 실행 결과 안정성
- **견고성 (Robustness)**: 입력 변동에 대한 내성
- **안전성 (Safety)**: 위험한 행동 감지
- 벤치마크별 신뢰성 차원 탐색 지원

### 기술적 특징

- **Weave 통합**: 에이전트 트레이스 + 비용 자동 로깅
- **로컬/클라우드 실행**: 평가를 로컬 또는 클라우드에서 완전 병렬화
- **통합 CLI**: 모든 벤치마크/에이전트에 대한 단일 명령줄 인터페이스
- **에이전트 어댑터**: 새 에이전트 추가를 위한 표준 인터페이스 제공

### 평가 파이프라인

```
Task Selection → Agent Execution → Trace Capture → Cost Tracking → Scoring → Leaderboard
```

각 단계에서 토큰 사용량, API 호출 비용, 실행 시간이 자동 기록됨.

## AHOY와 비교

### AHOY보다 나은 점

1. **다차원 신뢰성 평가**: 정확도 외에 일관성/견고성/안전성까지 체계적 측정. AHOY는 pass/fail 이진 판정만.
2. **비용-성능 트레이드오프 내장**: 평가에 비용을 기본 변수로 포함. AHOY는 토큰/비용 추적 없음.
3. **재현성 보장**: 동일 벤치마크에서 동일 조건으로 재현 가능한 평가. AHOY 평가는 LLM 응답의 비결정성으로 재현 어려움.
4. **멀티 벤치마크 통합**: 5개 이상 벤치마크를 단일 CLI로 실행. AHOY는 프로젝트별 ad-hoc 평가.
5. **에이전트 트레이스 자동 캡처**: Weave 기반 전체 궤적 로깅. AHOY는 트레이스 캡처 미구현 (T6-1 Agent Trace 제안 대상).
6. **학술적 검증**: ICLR 2026 수록 논문 기반. AHOY는 실무 도구로 학술 검증 없음.

### AHOY가 더 나은 점

1. **Generator-Evaluator 분리**: HAL은 "에이전트를 평가하는 하네스"이지 "에이전트 내부에서 생성-평가를 분리하는 하네스"가 아님. AHOY는 코드 생성 과정 자체에 적대적 평가를 내장.
2. **실시간 개입 (Hook 기반)**: HAL은 사후 평가(post-hoc). AHOY는 PreToolUse/PostToolUse Hook으로 **코드 생성 중** 실시간 개입.
3. **다중 모델 컨센서스**: HAL은 단일 에이전트를 벤치마크에서 평가. AHOY는 2개 이상 외부 모델의 합의로 편향 차단.
4. **파일 소유권/의견 제거**: HAL은 이런 세밀한 편향 차단 메커니즘이 목적이 아님. 다른 레벨의 도구.
5. **상태머신 기반 워크플로우**: HAL은 평가 파이프라인이지 개발 워크플로우가 아님. AHOY는 스프린트 상태머신으로 개발 전 과정을 관리.
6. **rework 루프**: HAL은 1회 평가 후 점수 부여. AHOY는 fail 시 자동 rework 3회까지 허용하여 품질 수렴.

### 비교 관점

HAL Harness와 AHOY는 **다른 레벨의 도구**:
- HAL = "에이전트 성능을 외부에서 측정하는 벤치마크 인프라"
- AHOY = "에이전트의 코드 생성 과정 내부에 적대적 검증을 삽입하는 하네스"

직접 경쟁보다는 **보완 관계**. HAL의 방법론을 AHOY 하네스 자체의 효과를 측정하는 데 활용 가능.

## 배울 만한 구체적 아이디어

### 1. 비용 추적 내장

HAL의 "비용을 기본 변수로" 접근은 AHOY에 직접 적용 가능. 현재 AHOY는 외부 모델 API 호출 비용을 추적하지 않음.

**적용 방향**: `eval_dispatch.py`에서 API 호출 시 토큰 수/비용 기록. 스프린트 완료 시 총 비용 리포트 자동 생성. rework 횟수와 비용 상관관계 분석 가능.

### 2. 신뢰성 다차원 평가

"정확도만이 아닌 일관성/견고성/안전성" 개념을 AHOY 평가에 적용:

**적용 방향**:
- **일관성**: 동일 이슈에 대해 평가 모델이 반복 실행 시 동일 결과를 내는지 확인 (공허 합의 감지 T4-4의 변형)
- **견고성**: 프롬프트 순서나 포맷 변경에도 동일 판정 유지 여부
- **안전성**: 평가자가 위험 코드(보안 취약점)를 놓치지 않는지 별도 검증

### 3. Weave 스타일 트레이스 로깅

HAL의 자동 트레이스 캡처를 AHOY의 T6-1 Agent Trace에 구체화:

**적용 방향**: JSONL 형식으로 `{timestamp, event_type, sprint_id, state_from, state_to, hook_result, eval_model, eval_result, tokens_used, cost}` 기록. 기존 `validate_harness.py`의 상태 전이 지점에 로깅 호출 삽입.

## AHOY 개선 제안 Top 3

### 1. 평가 비용 자동 추적 + 예산 시스템

> **v0.2.0 구현 완료** — `eval_dispatch.py:update_cost_tracking()` + `check_cost_limit()` eval_calls/tokens 누적 추적, Round 1/2 별도 기록, budget gate

**현재**: 외부 모델 API 호출 비용 비가시.
**제안**: `eval_dispatch.py`에 토큰 카운터 + 비용 계산기 추가. 스프린트당 평가 예산 상한 설정 가능 (`contract.md`에 `eval_budget_usd` 필드). 예산 초과 시 경고 또는 저렴한 모델로 자동 폴백.
**변경 대상**: `eval_dispatch.py` (API 응답에서 usage 파싱), `templates/spec_template.md` (예산 필드), `validate_harness.py` (예산 검증)

### 2. 평가 재현성 시드 도입

**현재**: LLM 평가의 비결정성으로 동일 코드에 대해 다른 평가 결과 가능.
**제안**: 평가 시 temperature=0 강제 + seed 파라미터 지정 + 동일 코드에 대한 평가 결과 캐싱. 동일 입력에 대해 재현 가능한 판정 보장.
**변경 대상**: `eval_dispatch.py` (API 호출 파라미터에 temperature=0, seed 추가)

### 3. 스프린트 트레이스 JSONL 로깅 (T6-1 구체화)

**현재**: 상태 전이/평가 결과의 체계적 로깅 없음.
**제안**: HAL의 Weave 통합에서 영감받아, 모든 상태 전이/Hook 실행/평가 호출을 `sprints/{id}/trace.jsonl`에 자동 기록. 각 이벤트에 타임스탬프, 토큰 사용량, 비용 포함. handoff 문서 자동 생성의 원천 데이터로 활용.
**변경 대상**: `validate_harness.py` (로깅 함수 추가), `eval_dispatch.py` (평가 결과 로깅)
